KinetAios vs Claude Code vs Codex

Three agent engines, one dashboard. Side-by-side comparison of architecture, capabilities, token efficiency and unique features — as of KinetAios v1.2.0.

01 At a Glance
Built-in Engine

Kaios

Native ReAct loop. GLM / Anthropic dual-protocol. Zero external CLI dependency — runs inside the Electron process.

9 built-in tools
  • SQLite + FTS5 full-text memory
  • MCP Bridge (local server + remote SSE)
  • Game-style Town View visualization
  • 4 themes · 4 languages i18n
CLI Subprocess

Claude Code

Anthropic's full CLI agent. Spawned via claude -p --output-format stream-json. Industry's most complete agent abstraction.

30+ tools & growing
  • Prompt cache (segmented + cache_control)
  • Recursive subagent dispatch
  • Five-level context compression
  • No OS-level sandbox
CLI Subprocess

Codex

OpenAI's Rust-powered agent. Spawned via codex exec --json. Only engine with true OS-level sandboxing.

Rust multi-crate
  • OS sandbox (Seatbelt / landlock / Win token)
  • Guardian automated review
  • WebSocket + HTTPS transport fallback
  • Heaviest abstraction layer

04 Strengths & Trade-offs

Kaios

Strengths
  • + Zero CLI dependency — runs in-process
  • + Smallest system prompt → cheapest per turn
  • + Pre-scans skills/MCP/CLI → fewer retry-wasted tokens
  • + Full GUI: Town View, 4 themes, 4 languages
  • + User-defined custom tools
  • + MCP Bridge: local server + remote multi-machine
Trade-offs
  • − 3-level compression (vs CC's 5-level)
  • − No streaming-while-executing tools
  • − No OS-level sandbox (confirm bridge only)
  • − Token estimation is approximate

Claude Code

Strengths
  • + Most complete agent abstraction (30+ tools)
  • + Best prompt-cache implementation
  • + 5-level context compression
  • + Full subagent suite (recursive/background/team)
  • + Stream + execute tools simultaneously
Trade-offs
  • − Heaviest system prompt → most expensive per turn
  • − No OS-level sandbox
  • − Requires Anthropic API key
  • − No custom user-defined tools
  • − CLI-only — no native GUI

Codex

Strengths
  • + Only engine with OS-level sandbox
  • + Guardian automated security review
  • + WebSocket + HTTPS transport fallback
  • + Stream + execute tools simultaneously
  • + Remote context compaction
Trade-offs
  • − Heaviest architecture (Rust multi-crate)
  • − Requires OpenAI API key
  • − No custom user-defined tools
  • − CLI-only — no native GUI
  • − Weakest in-app memory story

06 Only in KinetAios

Features that exist nowhere else — not in Claude Code CLI, not in Codex CLI.

Town View

Game-style isometric map: projects = houses, conversations = villagers, real-time agent status (Idle / Working / Done / Error). Chat inline on the map.

MCP Bridge (Multi-Machine)

Run a local MCP Server to accept remote run_agent calls. Connect to other machines via SSE. Your agents become a distributed cluster.

4 Themes & 4 Languages

Dark, Light, Aurora, Serene themes. English, Simplified Chinese, Traditional Chinese, Japanese — fully internationalized UI.

Custom Tools

Define your own tools in the UI — name, description, schema, handler. Instantly available to the Kaios engine alongside the 9 built-ins.

Three-Engine Dashboard

Switch between Kaios / Claude Code / Codex per conversation. Compare outputs side by side. All three share one unified event stream.

Memory Panel + Timeline

SQLite FTS5-backed memory with full UI: filter by channel, inline edit, delete, JSON import/export. Memory timeline with automatic decay.

Which engine should you use?

All three share an identical ReAct skeleton. The difference is entirely in execution safety, context management, and cost.

Save tokens & full visual control → Kaios Max completeness, budget OK → Claude Code Sandboxed secure execution → Codex
